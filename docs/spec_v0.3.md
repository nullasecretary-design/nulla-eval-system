## 一、專案背景與目標

### 1.1 為什麼要做這個系統

公司原本使用紙本 / Notion / Google Sheet 處理績效評核,但遇到以下問題:

- **權限控管不足**:同事間會互相看到評核資料
- **無法跨公司整合**:三個組織需要分別管理
- **沒有自動化**:評核提醒、報表都要手動處理
- **付費門檻**:Notion AI Plus 才能解決權限問題,但無預算

### 1.2 系統目標

打造一個 **PWA 網頁應用程式**(可加到手機主畫面、體驗近似原生 APP),整合三個組織的績效評核流程,實現:

- 員工自評 + 主管評核 + 執行長評核的三層評核流程
- 嚴格的權限控管(資料隔離、最小權限原則)
- 自動季度報表
- LINE 推播提醒(階段二實作)
- 跨公司統一管理介面

### 1.3 開發限制

- **預算**:零預算,只能用免費服務
- **開發者**:零經驗的非工程師(本人)+ AI 輔助(Claude Code)
- **時程預估**:8-12 週(約 2-3 個月)

## 二、組織架構與使用者角色

### 2.1 三個組織

| 組織 | 規模 | 結構 |
|------|------|------|
| Nulla 行銷公司(本公司) | 6 人 | 有主管階層(主管 → 下屬) |
| 診所 1 | ~10 人 | 扁平(員工 → 執行長) |
| 診所 2 | 30~40 人 | 扁平(員工 → 執行長) |
| (未來)新診所 | 待定 | 待定 |

**重要**:三家組織共用同一個系統,但資料完全隔離(多租戶設計)。

### 2.2 兩種屬性:職位 vs 管理者身分

> **v0.2 重要更新**:每個員工有兩個獨立屬性,分開設定。

#### 職位(評分視角)

決定員工在評核流程中的角色:

| 職位 | 範圍 | 評核權限 |
|------|------|---------|
| 一般員工 | 自己 | 看自己的評核、填自評、看歷史 |
| 主管 | 自己 + 下屬 | 員工權限 + 評下屬 + 看下屬資料 |
| 執行長 | 全公司 | 評所有人 + 看全公司資料 |

#### 管理者身分(管理視角)

獨立於職位,預設為「無」,只有超管可以指派:

| 管理者身分 | 權限說明 |
|-----------|---------|
| 無 | 預設值,大部分員工 |
| 秘書 | 看自家公司評核進度、解鎖自家評核、啟動本月評核、新增員工(基本資料) |
| 會計 | 看三家公司報表、產生報表、下載報表 |
| 超級管理員 | 後台所有功能、解鎖跨家評核、設定管理者身分 |

#### 兩個屬性的組合舉例

- **一般員工**:職位=一般員工,管理者身分=無 → 只能填自評
- **部門主管**:職位=主管,管理者身分=無 → 評下屬 + 看下屬
- **執行長**:職位=執行長,管理者身分=無 → 評全公司
- **公司秘書**:職位=一般員工,管理者身分=秘書 → 自己評核 + 後台行政功能
- **公司會計**:職位=一般員工,管理者身分=會計 → 看三家報表
- **Becca(本人)**:職位=主管,管理者身分=超級管理員 → 評下屬 + 後台全部

### 2.3 跨公司權限

以下三人需要看到所有三家公司的資料:

- **本人(系統建置者)** → 超級管理員
- **執行長**(同時擁有三家公司) → 預設只看自家,需切換單位才能跨家檢視
- **會計**(任職於執行長的另一家公司,負責三家薪水) → 管理者身分=會計,可看三家報表

實作方式:登入後預設進入主要單位,**右上角有「切換單位」按鈕**,只有上述高權限角色看得到此按鈕,一般員工不會看到。

## 三、評核規則

### 3.1 評核項目(共 100 分)

| 項目 | 滿分 |
|------|------|
| 工作時效 | 30 |
| 工作品質 | 25 |
| 工作配合度 | 25 |
| 出勤狀況 | 20 |
| **總分** | **100** |

### 3.2 評核頻率與時程

- **頻率**:每月一次
- **季度結算**:每 3 個月為一季,在每季末月(3、6、9、12 月)產生季度報表
- **季度結算用途**:影響下一季度的獎金

### 3.3 評核權重

#### 一般員工(有主管者)

- 員工自評:**僅供參考**(不計入總分,但必填,供主管對照認知落差)
- 主管評核:**70%**
- 執行長評核:**30%**

#### 直屬於執行長者(無中間主管)

- 員工自評:僅供參考
- 執行長評核:**100%**

### 3.4 評核時程規則

> **v0.2 重要更新**:取消「23 號自動寄通知」設計,改為人工觸發。

| 時間點 | 動作 | 對象 |
|--------|------|------|
| 每月 20 號起 | 後台首頁顯示「本月評核可啟動」提示 | 秘書 |
| 秘書按下啟動鈕 | 系統自動寄通知開始評核 | 全體員工(自動寄) |
| 員工填完自評 | 通知該員工的主管開始評 | 主管(自動寄) |
| 主管填完評核 | 暫不自動通知執行長,由秘書人工聯繫 | 秘書 → 執行長 |
| 截止前 2 天 | 未填名單由秘書手動寄提醒 | 未完成者(秘書手動寄) |
| 每月 1 號 | 確認執行長完成、數據正確,通知會計 | 本人(管理員) |

> 階段一:先做系統內提醒 + Email。階段二:接 LINE Messaging API 做推播。

### 3.5 通知策略哲學

> **v0.2 新增,重要原則**:

- **流程性自動通知**(員工填完→主管):自動寄,因為是「告知接力」,不會煩
- **催促性提醒**(未填者催促):人工觸發,避免月底訊息轟炸
- **啟動評核**:完全人工(秘書按按鈕),不自動,避免假日、颱風假、各公司節奏不同的問題

## 四、登入與身分驗證

### 4.1 主要登入方式:LINE Login + 員工編號綁定

> **v0.2 重要更新**:不再使用 LINE ID 比對(因為大部分人不知道自己的 LINE ID,且 LINE 沒強制要求設定),改為員工編號綁定。

#### 為什麼選 LINE 登入

- 完全免費(LINE Login 免費)
- 大家手機都有 LINE,不用記帳號密碼
- 免去簡訊驗證碼成本
- 之後評核提醒可直接用 LINE 推播,一石二鳥

#### 為什麼用「員工編號」綁定而非 Email

- 刀房護理師等現場員工沒有公司 Email
- 員工編號管理員可代發,口頭/便條紙告知即可
- 員工編號未來可串聯薪資、勞健保系統,一物多用
- 不需要員工知道自己的 LINE ID

### 4.2 員工編號規則

建議格式(可自訂):

- Nulla 行銷:`N001` ~ `N006`
- 診所 1:`C1-001` ~ `C1-010`
- 診所 2:`C2-001` ~ `C2-040`

### 4.3 登入流程

#### 第一次登入(綁定流程)

1. 員工打開網站
2. 點「使用 LINE 登入」
3. LINE App 跳出授權畫面
4. 同意授權
5. 系統提示:「請輸入你的員工編號完成綁定」
6. 員工輸入編號(例:N001)
7. 系統綁定成功 → 進入該員工首頁

#### 之後登入(無痛登入)

1. 打開網站 → 點 LINE 登入 → 直接進入首頁

### 4.4 換手機/LINE 帳號被盜的重綁機制

> **v0.2 重要更新**:LINE 帳號變更時的處理方式。

#### 一般員工(自助流程,需 Email)

- **必須由超級管理員親自在後台操作重綁,不可自助**
- 避免有人冒充「我換 LINE 了」騙秘書解綁高權限帳號

## 八、報表(v0.2 完整設計)

### 8.1 季度報表

#### 產出時機

- **手動觸發**:管理員按「+ 產生新報表」按鈕
- 非自動排程,避免依賴系統記憶力

#### 產出格式

- **PDF + Excel 雙產出**
- 使用者下載時自選格式

#### 計算方式

> **季度分數 = (月1加權總分 + 月2加權總分 + 月3加權總分) ÷ 3**

- 一般員工:每月加權 = 主管評 × 70% + 執行長評 × 30%
- 直屬執行長者:每月加權 = 執行長評 × 100%

#### 缺評處理

- 該月有評才計入,缺評月份不算
- 整季全部缺評者:報表上顯示「無資料」(非 0 分)
- 缺評月份在主表上顯示「—」,旁邊加 `*` 標示

### 8.2 報表結構:主表

#### 頂部資訊區

- 季度名稱、產出日期、所屬公司

#### 摘要區(兩個卡片)

- 全公司平均
- 缺評提醒(本季有 X 位員工缺評)

> v0.2 移除「最高/最低分」卡片(對小規模組織意義不大)。

#### 主表(每員工一列)

- 姓名、部門、月1分、月2分、月3分、季度平均、詳細按鈕
- **不展示四項目拆解**(進詳細頁才看)
- **不展示主管/執行長原始分**(進詳細頁才看)
- **不展示自評分數**(進詳細頁才看)
- 缺評月份顯示「—」,該員工季度分後加 `*` 標示

### 8.3 員工詳細頁

- 頂部摘要:員工資料 + 季度總分
- 每月拆解表:自評、主管(70%)、執行長(30%)、加權結果、落差欄
- **落差自動標示**:主管與執行長落差 ≥ 10 分時黃底警示
- 四項目平均條狀圖(該季三個月平均後)

### 8.4 報表權限

> **v0.2 更新**:秘書完全看不到報表(避免資料外流)。

| 角色 | 看報表 | 手動產生 | 下載 | 自動寄送(階段三+) |
|------|--------|---------|------|------------------|
| 員工 | ❌ | ❌ | ❌ | ❌ |
| 主管 | ❌ | ❌ | ❌ | ❌ |
| 秘書 | ❌ | ❌ | ❌ | ❌ |
| 執行長 | ✅ 自家 | ✅ 自家 | ✅ | 可設定 |
| 會計 | ✅ 三家 | ✅ 三家 | ✅ | 可設定 |
| 超級管理員 | ✅ 全部 | ✅ | ✅ | 可設定 |

### 8.5 報表歷史

- 所有產出過的報表**永久保留**(不自動刪除)
- 記錄欄位:季度、範圍(單一公司/合併)、產出時間、產出者
- 可篩選:年度、公司範圍
- 勞檢、稽核、紛爭時可隨時調閱

### 8.6 合併檢視

- 「+ 產生新報表」彈窗中,公司範圍可複選
- 勾 1 家 = 單一公司報表
- 勾 2~3 家 = 合併報表
- 會計常用合併,執行長/超管常用單一

### 8.7 安全策略

- 採「LINE 登入 + 角色權限」,不另設密碼
- 階段三可考慮「下載報表時二次驗證」
- 階段一不過度設計,實用優先

## 九、後台管理介面(v0.2 新增)

- A. 員工管理

### 9.1 A. 員工管理

#### 功能列表

- 員工列表(可搜尋、篩選公司/職位/狀態)

- Excel 批次匯入(帶預覽確認)
- 單筆編輯員工資料

- **秘書介面看不到「管理者身分」欄位**,無法設定他人權限

- 從介面層級防止「秘書自己升自己為會計」的內部威脅

#### 已完成名單

- 預設收合,點擊展開
- 展開後顯示完整時間戳

#### 提醒功能

- **一鍵寄提醒**給所有未完成者
- **單獨提醒**某一人(避免騷擾已完成的人)
- 階段一:Email 通知;階段二:LINE 推播

#### 啟動本月評核

- 每月 20 號起,後台首頁顯示「本月評核可啟動」綠色提示卡
- **秘書手動按按鈕觸發**
- 觸發後系統自動寄通知給該公司全體員工開始填自評
- 超管也有此權限但不主動提醒(避免被追進度)

#### 解鎖評核入口

- 獨立「解鎖評核」頁面,統一處理所有解鎖請求
- 可查看歷史解鎖紀錄
- 解鎖原因選填,不強制

### 9.4 D. 系統設定

- 啟動提醒日(預設 20 號)

#### 通知設定

#### 公司管理

- 公司基本資料

#### 資料匯出

- 全系統員工資料匯出(備份用)
- 全評核紀錄匯出

#### 系統 log 查閱

- 解鎖紀錄

- 報表產出紀錄

### 整體設計哲學

> **資料完整收集,但顯示精準分流**

- 員工原始分都存著,但員工本人只看加權後分數
- 主管/執行長每筆評分都記錄,但歷史頁不顯示原始分
- 解鎖紀錄完整 log,但秘書解鎖時不通知執行長
- 秘書能看評核進度,但看不到報表內容
- 流程性通知自動發,但「催促」要人工觸發

> 這就是資安界的「**最小權限原則(Principle of Least Privilege)**」——每個角色只看到他完成工作所需要的資訊,不多不少。

| 後端 / 資料庫 | Supabase(PostgreSQL) | 免費方案夠用、含登入系統、可搬遷 |
| 部署 | Vercel | 免費方案、與 Next.js 完美整合 |
| 版本控制 | Git + GitHub | 免費、業界標準 |
| 登入 | LINE Login + 員工編號綁定 | 免費、用戶體驗好 |
| 推播(階段二) | LINE Messaging API | 免費、用戶體驗好 |
| Email 通知 | Resend / SendGrid 免費方案 | 管理者變更通知用 |

### 11.2 為什麼選這個組合

- **零成本**:全部免費方案,6 人到 50+ 人規模都夠用
- **可搬遷**:Supabase 資料可匯出、Vercel 程式碼在 Git 上,不被綁死
- **現代化**:這是 2024-2026 年新手做網頁 APP 的主流選擇,網路教學多
- **安全性**:Supabase 建構在 AWS 上,等同企業級雲端

### 11.3 與老闆溝通說法

> 「我們用 Supabase 作為資料庫,它本身建構在企業級雲端平台上,安全性與 Google Cloud / Microsoft Azure 同級,而且免費額度夠我們用。」
